## 1. Executive summary

The Procurement Data Simulator is a standalone application that generates **reproducible, provenance-stamped synthetic procurement datasets** across three industry verticals (Healthcare, Higher Education, Manufacturing). It addresses the need for realistic, relational, scenario-rich procurement data for demos, internal research, fraud-detection training, forecasting pipelines, and teaching — without tying generation to the Django ORM.

Each "generate" action emits a single `.zip` bundle containing 11 relational tables as CSV, XLSX, and SQLite (with foreign keys intact), plus a YAML recipe, a SHA256 manifest, a data dictionary, sample queries, and a README. Given a recipe and seed, any run is byte-reproducible.

## 2. Background & problem

### 2.1 Situation

The repo previously shipped:
- Three industry profile DSLs (`scripts/_industry_profiles.py`) encoding category mixes, supplier tiers, seasonality, and policies.
- Two Django management commands that seed procurement data *directly into ORM models* (`scripts/seed_industry_data.py`, `scripts/seed_demo_data.py`).

### 2.2 Problems with the prior state

1. **ORM-coupled generation** forces consumers to bring up Django, Postgres, and migrations before they can get a dataset.
2. **Single-shot behavior** — generation writes to a live DB; no artifact a user can email, version, or audit.
3. **No provenance** — there is no way to say "here is the exact dataset X was trained on" because configuration is implicit in command-line flags.
4. **No scenario layering** — fraud patterns, supply-chain shocks, and consolidation dynamics can't be toggled; they'd have to be forked into new commands.
5. **No "what does this data look like" feedback loop** — users run the command and find out later whether their profile edits had the intended effect.

### 2.3 Strategic framing

Don't build yet another "CSV generator." Build a **simulator**: a studio-style app where users shape the data-generating process visually, stack composable scenario layers, see the effect live, and export a self-describing, reproducible bundle. The generator is table stakes; the *studio, scenarios, and provenance* are the differentiators.

## 3. Goals & non-goals

### 3.1 Goals

1. **Industry-aware generation** for Healthcare, Higher Ed, Manufacturing out of the box.
2. **Relational integrity** — PRs → POs → GRs → Invoices, Supplier ↔ Category, Contract ↔ PO all joined by stable IDs.
3. **Reproducibility as a contract** — any bundle can be regenerated byte-exactly from its `recipe.yaml` + seed.
4. **Composable scenarios** — ordered, per-scenario parameterized transforms that reshape the dataset (fraud, consolidation, shortages, shocks, maverick spend).
5. **Live preview** — 4 charts that redraw in <200 ms as the user edits a profile, giving tangible cause-and-effect feedback.
6. **Useful exports** — CSVs (pipelines), XLSX (humans), SQLite (ad-hoc queries), plus a data dictionary and ready-to-run queries.
7. **One source of truth** for profiles shared between the simulator and the Django seeder.

### 3.2 Non-goals

1. **Production data warehouse** — bundles are synthetic and demo-scale (≤ 200k transactions). Not a replacement for a real data platform.
2. **Multi-tenant SaaS hosting** — local Streamlit is the primary deployment; shared deployment is explicitly deferred.
3. **Arbitrary schema customization** — the 11-table schema is fixed; users can edit parameters, not add columns or entities.
4. **Currency / locale / i18n** — USD, US English only in v1.
5. **Realistic sub-graphs that don't exist in the profiles** — e.g., RFQ/RFP flow, payment runs, cash application. Out of scope.
6. **Security testing or adversarial data** beyond the scenario set. Red-team scenarios would live in a separate expansion.

## 4. Target users & use cases

| Persona | Primary goal | How the simulator helps |
|---|---|---|
| **R&D engineer demoing analytics** | A realistic dataset for a pitch/demo in an hour | Pick industry → generate bundle → load into Power BI or a notebook |
| **ML engineer training fraud models** | Ground-truth labeled data with controlled fraud injection | Stack `plant_fraud(rate=...)` → bundle includes `policy_violations.csv` with severities |
| **Data engineer testing pipelines** | Stress a new ETL with volume + edge cases | Generate 200k-row bundle with mixed statuses; test joins, nulls, amendments, exceptions |
| **Instructor / workshop author** | Teach procurement analytics without exposing real vendor data | Baseline bundle + data dictionary + sample queries |
| **Procurement analyst exploring "what-if"** | Model impact of a hypothetical consolidation or shortage | Scenario stack + live preview shows outcome before committing |

### User stories

- *As an ML engineer*, I can inject plant-level fraud at a specified rate so my classifier has a labeled training set.
- *As a data engineer*, I can re-create the exact dataset from last quarter by running the saved `recipe.yaml`.
- *As an analyst*, I can see how changing the pharmaceutical share from 32% to 45% affects the supplier Pareto in <1 second.
- *As a consumer of the bundle*, I can open `dataset.sqlite` and run `sample_queries.sql` to get oriented in 30 seconds.
- *As an auditor*, I can verify a bundle has not been tampered with by comparing file SHA256s to `manifest.json`.

## 5. Functional requirements

### 5.1 Industry profiles

- Three baselines available: `healthcare`, `higher-ed`, `manufacturing`.
- Each profile specifies: org name, 15–16 categories (spend share + log-normal amount μ/σ + named suppliers), tail-supplier templates, departments, cost-center prefix, payment terms, 12-month seasonality, and 4 spending policies.
- Profiles are declarative dicts in `scripts/_industry_profiles.py`; the simulator imports this module unchanged, preserving the "one source of truth" contract with Django.

### 5.2 Profile Studio (edit the data-generating process)

- **Donut chart of spend mix** (always visible) + **inline data editor** for spend share, amount μ, amount σ per category.
- **Seasonality bar chart** + **inline data editor** for 12 monthly multipliers.
- **Normalization** — shares and seasonality are rescaled at generation time (sum to 1.0; mean to 1.0). Editor shows live sum/mean feedback.
- **Reset-per-section** buttons to restore baseline.
- **Baseline summary card** always visible under the header so users can see the industry baseline even while they're editing away from it.
- **Dirty indicator** (pill in header) shows how many fields diverge from baseline.

### 5.3 Scenario stack

Ordered list of composable post-generation transforms. Order matters.

| Scenario | Parameters | Effect |
|---|---|---|
| `plant_fraud` | `rate` (0.0–0.20) | Duplicate invoices, round-amount bias, split PO patterns |
| `supplier_consolidation` | `degree` (0.0–1.0) | Reassigns fraction of tail-supplier transactions to tier-1 |
| `category_shortage` | `category`, `quarter` (1–4), `multiplier` (1.0–10.0) | Amplifies amounts in a target quarter for one category |
| `pandemic_shock` | `month` (YYYY-MM), `severity` (0.1–0.9) | Drops transactions and deflates remaining in target month |
| `maverick_spend` | `rate` (0.0–0.5) | Flips a share of contract-backed POs to off-contract |

UI: select from dropdown, add to stack, reorder with ↑/↓, remove. Each scenario's parameters are editable inline.

### 5.4 Live preview

- Updates on every edit. 4 Plotly charts over a 2 000-transaction sample:
  - Supplier spend Pareto (top 20 suppliers + cumulative %)
  - Monthly spend trend (line)
  - Benford first-digit distribution (observed vs. expected)
  - Category mix — target share vs. actual share (grouped bars)
- Preceded by a 4-metric strip: total spend, average transaction, top-10 supplier share, invoice exception rate.
- **Cached** via `@st.cache_data` keyed on `(profile_json, scenario_spec_json, seed, n_tx)`. Typical hit: <2 ms; miss: 30–50 ms.
- Wall time surfaced in a header pill so users feel the cache.

### 5.5 Generation settings

Surfaced in the **Settings** tab:

- **Identity**: org slug, org name (optional)
- **Reproducibility**: seed
- **Date range**: start date and end date (both `date_input`, constrained)
- **Row counts**:
  - Master "Transactions" slider (1 000–200 000)
  - **Auto-scale** (default) — P2P counts derived from fixed ratios with floors
  - **Manual** mode — editable table with individual counts per entity (contracts, PRs, POs, GRs, invoices, violations)
- **Generate Bundle** button — blocked when dates are invalid

### 5.6 Bundle generation & download

- Single click generates all artifacts in memory and writes a `.zip`.
- **Last Bundle** tab surfaces the download, the recipe YAML, the manifest JSON, and per-table row counts.
- Bundle name follows: `{industry}-{with-scenario1-scenario2-...|baseline}-{UTC-timestamp}.zip`.

## 6. UX requirements

### 6.1 Layout (top to bottom)

1. **Header bar** — title + segmented control for industry + status pills (`seed`, `rows`, `scenarios`, `preview ms`, optional `custom N edit(s)` pill).
2. **Baseline summary card** — faint gray card with baseline org name, category count, top-3 categories by share, top-2 seasonal peaks.
3. **Main area** — 3 columns side-by-side:
   - Profile Studio (donut + spend-mix editor + seasonality bar + seasonality editor)
   - Scenario Stack (add / reorder / remove)
   - Live Preview (4 charts, 2×2)
4. **Divider + tabs**: Settings | Last Bundle
5. **Streamlit's built-in top toolbar** is made transparent; padding is tuned so no content hides under it.

### 6.2 Visual system

- Light theme, primary color `#4C78A8` (the shared chart blue).
- Tabular numerics in status pills and summary card.
- Uppercase, letter-spaced section labels; tight letter-spacing on titles.
- One accent color used consistently; warnings in amber (`#FEF3C7`/`#FCD34D`).
- Max page width 1600 px to keep the three-column layout comfortable on ultra-wide monitors.

### 6.3 Interaction principles

- **Live feedback first** — anything the user changes should be reflected in the preview within the next render frame.
- **No destructive defaults** — switching industries clears profile overrides only; scenario stack persists.
- **No confirmation dialogs** — every action is cheap and reversible.
- **Keyboard accessibility** — Streamlit's native tab-navigation is sufficient for v1.

## 7. Data model

11 tables emitted per bundle. String IDs throughout.

| Table | PK | Notable FKs |
|---|---|---|
| `categories` | `category_id` (`CAT-###`) | — |
| `suppliers` | `supplier_id` (`SUP-#####`) | — |
| `transactions` | `transaction_id` (`TXN-########`) | → `categories`, `suppliers` |
| `contracts` | `contract_id` (`CT-{ORG}-{YEAR}-####`) | → `suppliers` |
| `contract_categories` | (`contract_id`, `category_id`) | M2M link table |
| `policies` | `policy_id` (`POL-###`) | — |
| `policy_violations` | `violation_id` (`VIO-######`) | → `transactions`, `policies` |
| `purchase_requisitions` | `pr_id` (`PR-{ORG}-{YEAR}-#####`) | → `suppliers`, `categories` |
| `purchase_orders` | `po_id` (`PO-{ORG}-{YEAR}-#####`) | → `suppliers`, `categories`, `contracts`, `purchase_requisitions` |
| `goods_receipts` | `gr_id` (`GR-{ORG}-{YEAR}-#####`) | → `purchase_orders` |
| `invoices` | `invoice_id` (`INV-{ORG}-{YEAR}-######`) | → `suppliers`, `purchase_orders`, `goods_receipts` |

Every column is documented in `data_dictionary.md` inside each bundle and in the canonical schema at `procurement_simulator/bundle/data_dictionary.py`.

## 8. Technical architecture

### 8.1 Package layout

```
procurement_simulator/
├── app.py                    # Streamlit entrypoint
├── profiles/                 # thin wrapper → scripts/_industry_profiles.py
├── generators/               # pure Python, DataFrame outputs
│   ├── categories / suppliers / transactions (numpy-vectorized)
│   ├── contracts / policies / policy_violations
│   ├── purchase_requisitions / purchase_orders
│   └── goods_receipts / invoices
├── scenarios/                # composable (dfs, params, rng) → dfs
│   ├── plant_fraud / supplier_consolidation
│   ├── category_shortage / pandemic_shock / maverick_spend
├── bundle/                   # writer (CSV+XLSX+SQLite+recipe+manifest+docs)
├── preview/                  # plotly: pareto, monthly_trend, benford, category_mix
└── studio/                   # plotly: spend_mix_donut, seasonality_bars
```

### 8.2 Key architectural principles

- **Library is pure Python.** `app.py` is the only module that imports Streamlit. The rest can be used headlessly from scripts or notebooks.
- **Generators are vectorized.** Transactions use `numpy.random.Generator.lognormal` and weighted `choice` instead of per-row Python loops. This is what makes sub-second preview possible.
- **Scenarios are post-generation transforms.** Input and output are both `dict[str, pd.DataFrame]`. Keeps profile mutation (UI) and dataset mutation (scenarios) separated.
- **Referential integrity is maintained by construction.** Each generator that introduces FKs draws them from the prior DataFrame's PK column, never inventing IDs.
- **Bundle writing is atomic.** All artifacts are built in memory and written to a single zip in one shot. Partial bundles cannot exist.

### 8.3 Dependencies

- Runtime: `pandas ≥ 2.0`, `numpy ≥ 1.24`, `pyyaml ≥ 6.0`, `openpyxl ≥ 3.1`, `streamlit ≥ 1.39`, `plotly ≥ 5.18`
- No new service-level dependencies (no DB, no message queue, no auth provider).

### 8.4 Integration with Django

- The simulator reads `scripts/_industry_profiles.py` directly. Changes to profiles flow to both the simulator and the existing Django management commands.
- The Django commands are unchanged in v1; a future pass can refactor them to call the new `procurement_simulator.generators` package and ORM-persist the output (one source of truth, two sinks).

## 9. Non-functional requirements

| Concern | Target | Current |
|---|---|---|
| **Preview latency** (cold) | < 250 ms on 2 000-txn sample | ~30–50 ms |
| **Preview latency** (cache hit) | < 10 ms | ~1.7 ms |
| **Full-bundle generation** (25k txn) | < 10 s | ~3–5 s |
| **Max bundle size** | 200 000 txn | enforced via UI slider cap |
| **Bundle memory footprint** | < 1 GB peak | well under for documented sizes |
| **Referential integrity** | 0 FK violations in SQLite | verified via `PRAGMA foreign_key_check` in smoke tests |
| **Reproducibility** | Identical output for same recipe+seed | verified via equality check on repeated runs |
| **Startup time** | < 5 s from `streamlit run` to first paint | ~2 s |
| **Cross-platform** | Windows + macOS + Linux | all supported (pure Python) |

## 10. Reproducibility contract

The tool's core promise.

### 10.1 What is captured in `recipe.yaml`

- Generator version (`0.1.0`)
- Industry key
- Full `GenerationConfig` (seed, row counts, date range, org identifiers)
- **Profile overrides only** — diffs vs. the baseline profile (keeps recipes small and meaningful)
- Ordered scenario stack with parameters

### 10.2 What is captured in `manifest.json`

- Bundle name, generator version, UTC generation timestamp
- Industry, seed, scenarios
- Per-table row counts
- Per-artifact SHA256 + byte-size for every file in the zip

### 10.3 Guarantees

1. **Byte equality** for all content-stable files (CSVs, SQLite, recipe, data dictionary, README, sample queries) when the recipe, seed, and generator version are identical.
2. **Timestamped artifacts** (`manifest.json`, bundle name) will differ run-to-run — these are documentation, not data.
3. **Manifest verification** — every consumer can re-hash a bundle's files and confirm against `manifest.json`.

```
    ├── data/

    │   ├── suppliers.csv
    │   ├── transactions.csv
    │   ├── contracts.csv
    │   ├── contract_categories.csv
    │   ├── policies.csv
    │   ├── policy_violations.csv
    │   ├── purchase_requisitions.csv
    │   ├── purchase_orders.csv
    │   ├── goods_receipts.csv
    │   └── invoices.csv
    ├── dataset.xlsx            # multi-sheet workbook
    ├── dataset.sqlite          # FKs enabled, queryable immediately
    ├── recipe.yaml             # reproducibility manifest
    ├── manifest.json           # SHA256 + row counts + timestamps
    ├── data_dictionary.md      # every column explained
    ├── sample_queries.sql      # 10 ready-to-run queries
    └── README.md               # loading into pandas / Postgres / Power BI
```

## 12. Success metrics

### 12.1 Shipping signals (meaningful at launch)

- All three industry profiles produce a bundle that passes `PRAGMA foreign_key_check` with zero violations.
- All five scenarios produce visibly distinct preview charts when enabled at a non-trivial rate.
- Round-trip test: same recipe + same seed → identical CSV bytes.

### 12.2 Usage signals (post-launch)

- Number of bundles generated per week (aggregate, local telemetry not in v1).
- Number of distinct scenarios-on-stack in generated bundles (indicates real use, not baseline-only).
- Anecdotal reports of bundles being used in demos / research / onboarding.

### 12.3 Quality signals (post-launch)

- Zero FK-violation reports from bundle consumers.
- Zero "why doesn't my recipe reproduce" reports after generator version pin.

## 14. Risks & mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Profile edits produce unrealistic data (e.g., one category at 95% share) | Medium | Low | Normalize at generation; preview reveals the oddness visually |
| Scenario stacking compounds unrealistically (fraud + maverick + shock) | Medium | Low | Scenarios are pure transforms; worst case the bundle looks extreme. Phase-4 warnings would call this out |
| Django profile changes break simulator silently | Low | Medium | Shared import of `scripts/_industry_profiles.py`; unit test to catch schema drift belongs in Phase 4 |
| Cache memory growth with many distinct edits | Low | Low | `@st.cache_data(max_entries=32)` caps memory; session-local |
| Streamlit version drift breaks `segmented_control` / `data_editor` | Low | Medium | `requirements.txt` pins `streamlit ≥ 1.39`; fallbacks can be added if needed |
| Large bundles (200k txn) exhaust memory | Low | Medium | UI slider caps at 200k; SQLite writer stages through in-memory then one disk backup; acceptable at target sizes |

## 15. Open questions

These deferred from the original design doc remain open in v1:

1. **Single-run vs. multi-run UX** — currently one generate ⇒ one bundle. Keep the "Bundle history" option (Phase 4E) in scope?
2. **Recipe versioning** — store recipes as source-controlled fixtures under `recipes/`? Today they're user-side only.
3. **Django integration** — keep the existing `scripts/seed_industry_data.py` wired to the new `procurement_simulator.generators` package? Currently the tools share *profiles* but not *generators*.
4. **Hosting** — local Streamlit only (current). Shared deployment (e.g., internal URL with run history) would need auth, per-user state, and persistence.
5. **Telemetry** — none in v1. Should we log bundle generations (anonymous count + scenarios used) to a lightweight telemetry endpoint to size Phase-4 investment?

## 16. Glossary

- **Profile** — the declarative dict for an industry (categories, seasonality, suppliers, departments, policies).
- **Scenario** — a post-generation, pure-function transform that mutates DataFrames to model a real-world phenomenon.
- **Scenario stack** — an ordered list of scenarios applied sequentially.
- **Recipe** — the `recipe.yaml` artifact capturing everything needed to reproduce a bundle (industry + overrides + scenarios + config).
- **Manifest** — the `manifest.json` artifact capturing generation metadata and SHA256 per file.
- **Bundle** — the single `.zip` containing the full reproducible dataset and its provenance.
- **Live preview** — the 4-chart strip that updates on every edit, powered by a 2 000-transaction sample and Streamlit's cache.
- **Baseline** — the unedited industry profile as defined in `scripts/_industry_profiles.py`.
